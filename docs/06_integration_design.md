## 1. なぜ統合が必要か(本リポ現行BTの構造的欠陥)

実装監査(2026-06-24)で判明。要件§9の拡張(LightGBM等)を積む前に**土台を直す**必要がある。

1. **エントリーのルックアヘッド**: シグナルは day t の終値(陽線・20日高値・当日出来高)で判定するのに、
   リターン起点が `Open(t)`(`src/backtest/engine.py` `_add_forward_returns`)。寄りでは当日終値は不明=
   **買えない値で測定**。実運用は翌日寄り入り → **BTは実運用より1日分かさ上げ**。
2. **市場/サイズ中立なし**: 生 Open-to-Open。finance 最大の教訓(検証10/11)=生リターンはサイズβを
   シグナルと誤認する。+0.35% は地合いβの疑い。
3. **銘柄ごとの in-sample 選別**: `src/screening/screener.py` は各銘柄の**自分の過去条件付きリターン**で
   足切り・ランク。同一5年標本での選別=過学習、しかも1銘柄サンプル薄。本来はパターンを全銘柄プールで測る。
4. **コスト無視**: +0.35%/日 狙いで手数料・スプレッド・寄りスリッページがゼロ。控除後ほぼ消える水準。
5. **生存バイアス**: 現プライム構成+yfinance。廃止/降格銘柄が欠落(finance 検証12と同根)。
6. **最も裁定が進んだ池**: プライム大型。finance が「流動・高カバレッジに簡単なエッジは残らない」と
   1年かけて実証した、まさにその土俵。

## 2. 設計思想の転換 — 予測単独 → 3層の意思決定支援

「翌日寄りで期待値が正の銘柄を当てる」(本リポ要件§1.1)を、**3層**に再定義する:

- **層A:候補生成(テクニカル期待値)** — 既存パターンを**正直に再測定**した上で候補を出す。エッジが
  無ければ「無し」と表示し縮退する(賭けない)。
- **層B:回避オーバーレイ(idea3)** — 触ってはいけない銘柄(行使価額修正条項付新株予約権/MSCB 等)を
  **ハード除外**。finance で唯一生き残った信号。無料の保険。
- **層C:レジーム/リスク管理** — 地合いに応じて新規・建玉サイズを絞る。方向当てではなく(§1遵守)、
  ドローダウン縮小を狙う。

**この設計の要点:層A(=本リポが賭けてきたテクニカル予測)が再検証で消えても、層B+Cで価値が残る。**
最も起こりうる結末(=層Aのエッジ消失)に対して頑健。

## 3. 物理統合方針

- **母体 = stock-screener(本リポ)**。製品殻・CI・Streamlit Cloud デプロイがあるため。
- finance からは**コードのコピーではなく「メソッド」を pandas ネイティブで移植**する。
  finance の実装は SQLite + イベント(disclosure_id)結合で、本リポの pandas/parquet + (ticker,date) と
  結合キーが違うため。**移植するのは思想と計算式**(翌営業日始値入り・中央値中立・分割グリッチ除外)。
- finance は**研究/参照リポとして残す**(EDINET 取得層・検証ドキュメントの一次情報源)。
- 新パッケージ構成(本リポ `src/` に追加):
  ```
  src/
    backtest/       # 既存。engine.py を §4 で置換
    overlay/        # 新規。avoidance.py(層B)/ regime.py(層C)
    ingest/         # 新規。financing_events.py(yanoshin タイトル分類。キー不要)
  ```

## 4. コア仕様:正直なリターン/検証エンジン(engine.py を supersede)

finance `src/backtest/returns.py` / `market.py` の契約を pandas に移植する。
**実装(2026-06-24)= `src/backtest/honest.py`**。engine.py は破壊的に消さず**素朴版との対比用に温存**し、
honest.py を正系統とする(`entry_shift=0` で旧来の同日寄り入りも再現でき、ルックアヘッドの寄与を可視化)。
取得は `scripts/fetch_prices.py`(adj_close 保持・再開可能)、実行は `scripts/run_honest_backtest.py`。

### 4.1 エントリーとリターン
- シグナル日 = day t(当日 OHLCV で判定)。**エントリー = `Open(t+1)`**(翌営業日始値)。
  本リポ実運用(夜間スクリーニング→翌寄り)と一致させる。
- `R(n) = adjOpen(t+1+n) / adjOpen(t+1) - 1`。始値は `adj_open = open * (adj_close/close)` で分割調整。
- 保有窓内に日次 ±50% 超(0.5〜2.0倍外)の急変があれば**未調整分割/不良とみなし当該イベントを除外**
  (finance `SPLIT_LOW/HIGH`)。

### 4.2 市場/サイズ中立(最重要の移植)
- ベンチ = **対象ユニバースの日次クロスセクション中央値リターンの複利**(finance `build_market_index`)。
  大型TOPIXではなく**自ユニバースの中央パス**を使うのが肝(サイズβを自動で吸収)。
- 出力は **raw と 中立後(adj)を両方**保持。判定は adj を主に読む。

### 4.3 プール集計 + 期間外検証
- per-ticker をやめ、**パターン(資金流入シグナル)が出た全銘柄×全日をプール**して期待値を測る。
- **ウォークフォワード**: 閾値・採否は学習期間で決め、**直後の期間外(OOS)**で測る(例:5年を
  学習4年→検証1年でローリング)。in-sample 選別を禁止。
- 出力統計は既存踏襲(中立後の重み付き中央値リターン・勝率・DD中央値)に **OOS フラグ**を追加。

### 4.4 コスト控除
- `settings.yaml` に `costs: {commission_bps, spread_bps, slippage_bps}` を追加。
- 期待値は**往復コスト控除後**を正とする。寄り成り入りの逆選択を見込み slippage を保守的に置く。

### 4.5 合否(層Aを採用するか)
- **GO**: 中立・OOS・コスト控除後に正の期待値が残り、季節(四半期)で反転しない。
- **NO-GO**: 上記で消える(finance の実証からは濃厚)→ 層Aは「候補補助」に格下げし、**層B+Cを主**にする。
  この結果も正直にドキュメント化(null を記録)。

## 5. 層B仕様:回避オーバーレイ(idea3 移植)

finance 検証10 で**本プロジェクト唯一の頑健な信号**(中立後 +60日 ~-3〜-4%、2.5年9/11四半期で負、
ただし**回避専用・ショート不可・レジーム依存**)。

- **イベント源**: yanoshin TDnet タイトル分類(キー不要。finance `classify_financing`)。
  - Tier A(除外対象)= **行使価額修正条項付の新株予約権 / MSCB**。
  - 普通の第三者割当"増資"は劣後しない → **除外しない**(過剰フラグ回避)。
- **適用**: Tier A 発表から N 営業日(既定60)以内の銘柄を**候補からハード除外**。
- **UI**: 候補外でも銘柄詳細・保有管理で**赤旗バッジ**を表示(どこで遭遇しても人間が避けられる)。
- **EDINET 拡張(任意・後続)**: ディスカウント率・希薄化率・GC注記まで精緻化はローカルLLMの本命用途。
  初版はタイトル分類のみ(無料・キー不要・即実装)。
- **棄却条件**: 除外オーバーレイを任意ロング母集団に乗せても DD/リスク調整後が改善しない場合は層Bを軽量化。

## 6. 層C仕様:レジーム/リスクオーバーレイ(finance 未着手MVP)

- **指標**: `^N225` × `^GSPC` の転がり相関 + 単純なリスクオフ判定(例:`^GSPC` が50日線割れ)。
- **使い方(方向当てに使わない=§1遵守)**:
  - 高相関リスクオフ時は**新規スイングを抑制 / 建玉サイズを縮小**。
  - 保有管理タブに**地合いバナー**(リスクオン/オフ)と推奨サイズ係数を表示。
- **検証(MVP)**: 任意ロング母集団に「高相関リスクオフ時は新規停止」を適用し、**最大ドローダウンが
  縮むか**を過去で確認。無料(yfinance)・最速で白黒。
- **棄却条件**: DD が縮まない/単なる遅延β → 層C不採用。

## 7. データ層と既知の限界

- 維持: yfinance(プライム日足5年)+ プライムマスタ + GitHub Release parquet ストア(製品インフラ)。
- 追加: yanoshin(タイトルメタ・キー不要)= 層Bのイベント源。EDINET は後続拡張(キー要)。
- **絶対ルール1(取得層の抽象化)を継承**: ソースはインターフェース越し(非公式API依存のため差し替え可能に)。
- **生存バイアス(正直に明記)**: 層AのBTは現プライム構成のため廃止/降格銘柄を欠く。無料データでは
  完全には埋まらない(finance 検証12)。→ 層Aの数値は**やや楽観側**と注記して読む。層B(除外)には影響軽微。

## 8. UI / CI への反映(製品殻の拡張)

- メイン画面: 候補表に **(a) 中立後OOS期待値・コスト控除後**、**(b) 赤旗列(層B)**、**(c) 地合いバナー(層C)** を追加。
- 銘柄詳細: 日数別期待値は**中立後**で表示。赤旗理由(Tier A 発表日)を併記。
- 保有管理: 地合いに応じた推奨サイズ係数・新規可否を表示。
- GitHub Actions 夜間バッチ: 価格更新に加え **yanoshin イベント更新 + レジーム状態更新**を追加。

## 9. ガバナンス(finance 絶対ルールの継承)

1. 取得層はインターフェース抽象化(§7)。
2. LLM を使う層(層B精緻化等)は **prompt_version / model_name を必ず記録**。
3. バックテストはルックアヘッド・チェックリスト(finance design §6.3)を毎回適用。起点=翌営業日始値。
4. 外部APIはレート配慮(yanoshin はスリープ・夜間分散)。
5. 結果は **null も含めて正直に記録**。設計と食い違えば本書を更新してから実装。

## 11. 各層の損益貢献の正直な見立て

- **層B(回避)= システムの核に確定**(検証09)。ユニバースを小型へ拡張したところ、Tier A の88%は小型に在り、
  **小型 Tier A は中立後 +20日 -5.2% / +60日 -7.7%・45%が10%超下落**(finance idea3を再現・凌駕)。
  生存バイアスで真の劣後はさらに深い=回避用途には保守的。**無料・コスト0で左裾を削る、最も確度の高い価値。**
- **層C(リスク管理)= GO 確定**(検証10)。単純な risk-off フィルタ(GSPC<50日線で新規を控える)で
  **最大DDを約-21%→-7%に圧縮、リターンも増加**(β縮小でない、r/DD 3.5→20)。§1遵守(方向でなくリスクゲート)。
  ただし便益の大半は2022の1エピソード依存=倍率でなくDD抑制効果を信頼。**高相関の絞りは逆効果=単純トレンドが本命。**

### 段A+段B+ユニバース拡張の確定結論(2026-06-24、検証07-09): **これは「回避ツール」である**
- **層A(テクニカル期待値)= 全セグメントで不採用。** 小型は明確に負(ブレイク=反転、n=20 -2.06%・全6年負)、
  Prime は実質ゼロ(見かけの+0.47%は混合ベンチのサイズβ)。素朴版の+3〜+4%は丸ごとルックアヘッド。
- **層B(回避)= 採用、核。** 小型 Tier A は -5〜-8%で頑健。
- → **本システムは「ブレイク候補を買う」スクリーナーではなく、「小型の地雷を回避する意思決定支援」**。
  予測でなく回避=「負けにくさも価値」に完全一致。次は層Bの運用化(全ユニバース赤旗+除外)→層C→UI/CI(段D)。
- **層A(テクニカル期待値)**: ~~再検証まで未証明~~ → **2026-06-24 再検証で NO-GO 確定(検証07)**。
  N=7,312・中立+OOS+コスト控除後、最良ホライズンでも中央値 -0.04%(基準+0.35%未達)。素朴版の+3.35%は
  丸ごとルックアヘッドだった。**候補補助に格下げし、賭けない。**

→ 最適なシステムとは「層Aが当たること」を祈る設計ではなく、**層Aが外れても層B+Cで成立する設計**である。
**段Aの結果がまさにそれを裏付けた**:賭けの本命だった層Aが消えても、設計は層B+Cで成立する。次は段B(回避)。
